# Лабораторная работа №2, вариант 1400

Этот каталог содержит не только готовый отчёт, но и всю рабочую кухню,
нужную для воспроизведения результата. Ниже кратко и по-человечески
описано, что здесь лежит и как этим пользоваться.

## Зачем всё это разложено по папкам

У задачи есть две разные стороны:

1. Нужно воспроизвести историю ревизий в `git` и `svn`.
2. Нужно оформить сдаваемый отчёт.

Чтобы не смешивать технические артефакты с отчётом, рабочая часть и LaTeX
разнесены по разным каталогам.

## Как это работает по шагам

### 1. Откуда берутся ревизии

Блок-схема задаёт только структуру истории: где идут обычные коммиты,
где создаются ветки и где происходят слияния.

Содержимое каждой ревизии задаётся архивами из `lab/input/commits/`. То есть:

- `commit0.zip` соответствует `r0`;
- `commit1.zip` соответствует `r1`;
- ...
- `commit14.zip` соответствует `r14`.

Скрипты не пытаются "додумать" изменения между ревизиями. Они каждый раз
приводят рабочее дерево к точному состоянию очередного архива.

### 2. Что делают скрипты в `lab/scripts/`

- `build_git.sh`
  Создаёт локальный Git-репозиторий, проходит по веткам схемы варианта,
  переключает пользователей `red` и `blue`, делает два merge и формирует
  итоговый граф.

- `build_svn.sh`
  Создаёт локальный SVN-репозиторий, настраивает `trunk/branches`,
  воспроизводит те же состояния и делает эквивалентные слияния в рабочей
  копии.

- `common.sh`
  Общие функции: применение снимков из архивов, очистка рабочего каталога,
  настройка пользователей и вспомогательные операции для Git и SVN.

### 3. Что считается результатом

После запуска скриптов основными артефактами являются:

- `lab/out/git-repo`
  Готовый Git-репозиторий с историей варианта 1400.

- `lab/out/svn/repo`
  Локальный SVN-репозиторий.

- `lab/out/svn/wc`
  Рабочая копия SVN, на которой воспроизводилась история.

- `lab/out/git-graph.txt`
  Текстовый граф Git для быстрой проверки.

- `lab/out/svn/log.txt`
  Лог SVN для проверки последовательности ревизий.

## Как что пересобрать

### Пересобрать техническую часть

```bash
./lab/scripts/build_git.sh
./lab/scripts/build_svn.sh
```

## Что важно помнить

- В отчёте должна быть только содержательная часть для проверяющего:
  задание, команды, ревизии и выводы.
- Внутренняя организация каталогов, отдельные репозитории и служебные скрипты
  нужны нам для удобства, воспроизводимости и совместной работы.
- В SVN есть служебные ревизии для `svn mkdir` и `svn copy`, поэтому его
  внутренние номера не совпадают один в один с узлами блок-схемы.
